# タスク・ギフト・送信機能の現状分析と不整合リスト

## 1. データ構造の定義（Firestore）

### 1-1. tasks コレクション（Firestore）

| フィールド名（Firestore） | 型 | 必須 | 説明 |
|---------------------------|-----|------|------|
| id | string | ○ | タスクID（ドキュメントIDと同一） |
| title | string | ○ | タスク名 |
| sender_id | string | ○ | 送信者UID |
| receiver_id | string | ○ | 受信者UID |
| status | string | ○ | "pending" / "active" / "pending_approval" / "completed" |
| reward_id | string | ○ | 紐づくギフトのドキュメントID（＝ gift.id） |
| gift_name | string | - | ギフト名（表示用） |
| target_days | number | ○ | 目標日数（1〜30） |
| current_count | number | ○ | 完了報告の累計回数 |
| last_completed_date | timestamp | - | 最終完了日時 |
| sender_name | string | - | 送信者表示名 |
| sender_emoji | string | - | 送信者アイコン絵文字 |
| sender_total_completed_count | number | - | 送信者の累計達成数 |
| completion_image_url | string | - | 完了報告画像URL（Storage） |

**書き込み元**: `TaskRepository.sendTask`（送信時）、`reportTaskCompletion`（完了報告時）、`approveTaskCompletion` / `rejectTaskCompletion` / `acceptReceivedTask`（承認・差し戻し・受け入れ時）。

---

### 1-2. gifts コレクション（Firestore）

| フィールド名（Firestore） | 型 | 必須 | 説明 |
|---------------------------|-----|------|------|
| id | string | ○ | ギフトID（ドキュメントIDと同一） |
| name | string | ○ | ギフト名 |
| is_unlocked | boolean | ○ | 解禁済みか |
| associated_task_id | string | ○ | 紐づくタスクID（＝ task.id） |

**書き込み元**: `TaskRepository.sendTask`（送信時に task と同時作成）、`approveTaskCompletion`（is_unlocked: true に更新）。

**注意**: アプリ内の `Gift` モデルは `title` だが、Firestore では `name`。受け入れ時にローカルで `Gift(title: dto.giftName)` のようにマッピングしている。

---

### 1-3. タスクとギフトの紐づき

| 紐づき | 内容 |
|--------|------|
| task → gift | `task.reward_id` === `gift.id`（同一ドキュメントID） |
| gift → task | `gift.associated_task_id` === `task.id` |

送信フロー（`sendTask`）で **task** と **gift** を同時に 1 件ずつ作成し、上記で一対一対応している。

---

### 1-4. 送信者・受信者の定義

| フィールド | 定義 | 備考 |
|------------|------|------|
| sender_id | タスクを送ったユーザーのUID | Firestore の tasks に必須。送信時に `Auth.auth().currentUser?.uid` を設定。 |
| receiver_id | タスクの送信先ユーザーのUID | Firestore の tasks に必須。送信時に SendTaskView で入力した UID を設定。 |

**結論**: Firestore の「送信タスク」では `sender_id` / `receiver_id` は正しく定義・保存されている。  
**自分用タスク**（AddTaskView で作成）は **Firestore に保存されず**、ローカル（UserDefaults）のみで `receiver_id` に相当するフィールドは存在しない。

---

## 2. 画面ごとの処理内容

### 2-1. タスク作成・送信の対応関係

| 画面・操作 | 実装クラス/View | 保存先 | sender_id | receiver_id | reward_id / ギフト |
|------------|------------------|--------|-----------|-------------|---------------------|
| **自分用タスクの作成** | AddTaskView → TaskViewModel.createTask | UserDefaults（tasks 配列） | なし（nil） | なし（プロパティなし） | なし（rewardDisplayName のみ任意） |
| **他ユーザーへタスク送信** | SendTaskView → TaskRepository.sendTask | Firestore（tasks + gifts） | ○ 現在ユーザーUID | ○ 入力した相手UID | ○ task.reward_id = giftId、gift 同時作成 |

「タスク作成」が **自分用** と **送信用** の 2 系統あり、保存先とフィールドが異なる。

---

### 2-2. ギフト作成の対応関係

| 画面・操作 | 実装 | 保存先 | Firestore gifts | タスクとの紐づき |
|------------|--------|--------|------------------|------------------|
| **自分用ギフトの作成** | AddGiftView → GiftViewModel（createGift / addGift） | UserDefaults（gifts 配列） | 書かない | unlockCondition.targetIds に taskId を指定可能（ローカルのみ） |
| **タスク送信時のギフト** | SendTaskView → TaskRepository.sendTask | Firestore（gifts 1 件 + tasks 1 件） | ○ 作成 | task.reward_id ↔ gift.associated_task_id |

自分用ギフトは Firestore に存在しない。送信時にだけ Firestore の gift が作成される。

---

### 2-3. 送信処理で書き込まれるフィールド（Firestore）

| 対象 | 書き込まれるフィールド | 通知に関連する項目 |
|------|------------------------|---------------------|
| tasks | id, title, sender_id, receiver_id, status, reward_id, gift_name, target_days, current_count, sender_name, sender_emoji, sender_total_completed_count | sender_id / receiver_id は Cloud Functions が FCM 送信先のユーザー取得に利用 |
| gifts | id, name, is_unlocked, associated_task_id | 通知には未使用 |

**users コレクション**: 送信処理では **直接は書き込まない**。  
`fcm_token` は **AuthManager**（ログイン時・FCM トークン更新時）で `users/{uid}` に保存される。  
通知を送るには、送信先・送信元の両方で **users に fcm_token が保存されていること** が前提。

---

### 2-4. 通知で参照している箇所

| タイミング | 実装箇所 | 取得している情報 | 通知先 |
|------------|----------|------------------|--------|
| タスク送信時（受信者へ） | SendTaskView.sendTask → fetchOtherUserProfile(uid: receiver) | receiver の fcm_token, display_name | 受信者（NotificationService.notifyTaskReceived は **ローカル通知** のみ） |
| 完了報告時（送信者へ） | TaskViewModel.completeTask → fetchOtherUserProfile(uid: senderId) | 送信者の fcm_token | 送信者（同上 **ローカル通知**） |
| 承認時（受信者へ） | Cloud Functions onTaskUpdated | Firestore users/{receiver_id}.fcm_token | 受信者（**FCM プッシュ**） |
| 完了報告時（送信者へ） | Cloud Functions onTaskUpdated | Firestore users/{sender_id}.fcm_token | 送信者（**FCM プッシュ**） |

アプリ内の「通知」はローカル通知、実際のプッシュは Cloud Functions が Firestore の `tasks` 更新と `users` の `fcm_token` を参照して送信している。

## 3. 明白な不整合（ギャップ）の抽出

| # | 不整合の内容 | 影響 |
|---|----------------|------|
| 1 | **「タスク作成」が 2 種類ある**（自分用＝ローカルのみ／送信用＝Firestore）。テスト時に「タスク作成」と言ったときにどちらを指すか曖昧。 | 通知テストでは「送信フロー」のみが Firestore と連動するため、送信〜受け入れ〜完了〜承認の流れで確認する必要がある。 |
| 2 | **自分用タスクには receiver_id が存在しない**（Task モデルに receiverId プロパティなし）。送信専用フローでは sendTask で receiver_id を必ず渡している。 | 設計上は一貫しているが、「タスク作成画面」だけでは通知テストに必要な Firestore タスクが作られない。 |
| 3 | **ギフトも 2 系統**（自分用＝ローカル／送信時＝Firestore）。AddGiftView で作ったギフトは Firestore にない。 | 通知テストには影響しない。送信フローで task + gift が同時に作成されるため、承認時の is_unlocked 更新は問題ない。 |
| 4 | **タスク送信時の「受信者への通知」**は NotificationService.notifyTaskReceived で **ローカル通知** を発火しているだけ。FCM で受信者へプッシュする処理は **Cloud Functions に未実装**（onTaskUpdated は pending_approval / completed のみ）。 | 「送信したら相手にプッシュが届く」をテストするには、Cloud Functions に「tasks ドキュメント作成（status=pending）時に受信者へ FCM」を追加するか、現状どおり「完了報告・承認」の 2 種類のプッシュのみテストする必要がある。 |
| 5 | **fcm_token の取得**は、送信時・完了報告時に `AuthManager.fetchOtherUserProfile(uid)` で行っている。Cloud Functions は Firestore の `users/{uid}.fcm_token` を参照。 | 両方とも同じ users ドキュメントを参照しており、ログイン/トークン更新で fcm_token が保存されていれば不整合はない。 |
| 6 | Firestore の **gifts** は `name`、Swift の **Gift** は `title`。 | 受け入れ時に giftName → title でマッピング済み。送信時は giftName をそのまま gift.name に保存しているため実害は小さい。 |

---

## 4. 通知テストを成功させるために最低限共通化すべきフィールドと修正案

### 4-1. 現状で揃えられている点（そのままでよい）

- Firestore の **tasks**: `sender_id`, `receiver_id`, `reward_id`, `status` は送信・完了報告・承認で一貫して利用されている。
- Firestore の **gifts**: `id`, `associated_task_id`, `is_unlocked` は送信と承認で一貫している。
- **users**: `fcm_token` は AuthManager で保存され、Cloud Functions が参照する形で共通。

### 4-2. 通知テストを成功させるための「最低限」の共通化・確認事項

| 項目 | 推奨対応 | 優先度 |
|------|----------|--------|
| **送信フローのみを「通知テスト用タスク」として扱う** | 通知テスト手順を「SendTaskView で送信 → 受信側で受け入れ → 完了報告 → 送信者で承認」に固定し、ドキュメント化する。AddTaskView で作るタスクは「通知対象外」と明記する。 | 高 |
| **users の fcm_token が必須であることを明示** | 通知が届かない場合のチェックリストに「送信者・受信者ともに一度ログインし、設定で通知許可し、Firestore の users/{uid} に fcm_token が保存されているか」を入れる。 | 高 |
| **タスク送信時の受信者向け FCM（任意）** | 「送信した瞬間に相手にプッシュを届けたい」場合は、Cloud Functions に `onDocumentCreated` で `tasks` の新規作成を検知し、`receiver_id` の `fcm_token` 宛に「〇〇さんからタスクが届きました」を送る処理を追加する。現状のままでも「完了報告」「承認」の 2 種類のプッシュはテスト可能。 | 中 |
| **用語の整理** | 「タスク作成」＝自分用（AddTaskView）、「タスク送信」＝他ユーザー向け（SendTaskView）と、ドキュメント・コメントで使い分ける。 | 低 |

### 4-3. 具体的な修正案（最低限）

1. **テスト手順の明文化（コードまたは docs）**  
   - 「通知テスト」では **SendTaskView でタスクを送信** し、受信側で受け入れ → 完了報告（画像あり可）→ 送信側で承認、の順で実行する。  
   - このとき Cloud Functions の **pending_approval**（送信者へ）と **completed**（受信者へ）の 2 種類のプッシュが発火することを確認する。

2. **users の fcm_token 前提の明記**  
   - 両端末でアプリ起動・ログイン済み、通知許可済み、かつ Firestore の `users/{送信者UID}` と `users/{受信者UID}` に `fcm_token` が入っていることとする。  
   - 未設定の場合は「通知が届かない」とし、AuthManager のトークン保存ログや Firestore で確認する手順をドキュメントに書く。

3. **（任意）タスク送信時の FCM**  
   - 必要なら Cloud Functions に、`tasks` の **create** トリガーを追加し、`status === "pending"` かつ `receiver_id` がある場合に `users/{receiver_id}.fcm_token` 宛に「〇〇さんからタスクが届きました」を送る。  
   - これにより「送信 → 受信者にプッシュ」も同じ Firestore / users のデータでテストできる。

## 5. まとめ表（データ構造・画面・不整合の対応関係）

| 機能 | データの流れ | 通知との関係 |
|------|--------------|--------------|
| 自分用タスク作成 | AddTaskView → TaskViewModel.createTask → UserDefaults のみ | 通知対象外（Firestore にタスクがない） |
| 自分用ギフト作成 | AddGiftView → GiftViewModel → UserDefaults のみ | 通知対象外 |
| タスク送信 | SendTaskView → TaskRepository.sendTask → Firestore tasks + gifts（sender_id, receiver_id, reward_id あり） | 受信者向けプッシュは現状ローカル通知のみ。FCM は Cloud Functions 追加で対応可能。 |
| 受け入れ | ReceivedBoxView → acceptReceivedTask → Firestore task.status=active、ローカルに Task + Gift 追加 | 送信者へ notifyTaskAccepted（ローカル）。 |
| 完了報告 | TaskListView 等 → completeTask → reportTaskCompletion → task.status=pending_approval | Cloud Functions が sender_id の fcm_token に FCM 送信。 |
| 承認 | ApprovalPendingView → approveTaskCompletion → task.status=completed, gift.is_unlocked=true | Cloud Functions が receiver_id の fcm_token に FCM 送信。 |

**通知テストを成功させるために最低限共通化すべきなのは「どのフローをテストするか」の認識と、送信・受信の両方で `users` に `fcm_token` が保存されていることの確認である。**  
Firestore の tasks / gifts のフィールドは送信フローで既に統一されているため、追加でフィールドを共通化するより、上記の手順と前提の明文化が有効です。
